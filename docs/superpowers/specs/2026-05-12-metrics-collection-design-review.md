# Metrics Collection Design Review

**Reviewed document:** `docs/superpowers/specs/2026-05-12-metrics-collection-design.md`  
**Review date:** 2026-05-12  
**Review type:** Design review  
**Status:** Changes requested

## Summary

The design direction is reasonable: an append-only local event log plus a query tool is a good fit for SagTask's existing local-first architecture. However, several details need correction before implementation. The highest-risk issues are tool registration coverage, unsupported execution-time tracking claims, non-idempotent subtask metrics, and the decision to keep `.sag_metrics.jsonl` git-tracked.

## Findings

### High: `sag_task_metrics` May Not Be Registered

The file inventory adds `handlers/_metrics.py` and updates `src/sagtask/__init__.py`, but actual tool dispatch is controlled by `src/sagtask/handlers/__init__.py` and its `_tool_handlers` map. `register()` iterates schemas and skips any tool without a handler in that map.

**Required fix:** Update the design to include `src/sagtask/handlers/__init__.py` changes:

- Import `_handle_sag_task_metrics`.
- Add `"sag_task_metrics": _handle_sag_task_metrics` to `_tool_handlers`.
- Add `_handle_sag_task_metrics` to `__all__`.

### High: Execution-Time Tracking Is Claimed But Not Designed

The spec scope includes “execution time tracking,” but the event model does not include start/end events or duration fields. `step_advance` timestamps alone cannot calculate the first step's start time, active duration, or pause-adjusted duration.

**Required fix:** Either remove execution-time tracking from scope or define explicit timing semantics. If keeping it, add events such as `step_enter`, `step_exit`, and `duration_ms_active`, and specify whether pause/resume intervals are excluded.

### High: Throughput Counting Is Not Idempotent

The design counts `subtask_complete` events by status, but `sag_task_plan_update` allows repeated and reversible status changes such as `done -> failed`, `failed -> done`, or repeated `done`. Counting raw events will double-count and can diverge from the current plan state.

**Required fix:** Add `old_status` and `new_status` to subtask events and aggregate by latest `subtask_id` state, or emit terminal events only on first transition from non-terminal to terminal. Document how reopened or corrected subtasks are handled.

### Medium: Git-Tracking `.sag_metrics.jsonl` Needs Reconsideration

The spec says `.sag_metrics.jsonl` is git-tracked. Metrics events include verification commands and pause reasons, which may expose local paths, environment details, or accidental secrets. Because `sag_task_advance` runs `git add -A`, routine metric writes will also create commit noise.

**Recommended fix:** Default `.sag_metrics.jsonl` to git-ignored unless there is a strong audit requirement. If it must be tracked, define redaction rules, commit timing, and branch conflict handling.

### Medium: `_emit_metric` Ownership Is Ambiguous

The helper is proposed in `_utils.py` as `_emit_metric(task_id, event, **fields)`, but writing `<task_root>/.sag_metrics.jsonl` requires resolving the task root through `SagTaskPlugin`. A utility with hidden provider access would make dependencies less clear.

**Recommended fix:** Prefer a `SagTaskPlugin.emit_metric(...)` method, or make the helper accept `task_root` explicitly. Keep path ownership close to the plugin object.

### Low: Coverage Parsing Is Too Fragile

The design only parses stdout when the command string contains `"cov"` using `r"TOTAL\s+.*?(\d+)%"`. This can miss coverage emitted to stderr, generated by wrapper scripts, or stored in XML/JSON reports.

**Recommended fix:** Parse stdout and stderr together, and consider optional step-level configuration such as `coverage_regex` or `coverage_report`. At minimum, document that coverage extraction is best-effort.

## Open Questions

- Is `.sag_metrics.jsonl` intended as an audit artifact or local runtime state?
- Should execution time mean wall-clock time, active unpaused time, verification command runtime, or subtask lifecycle time?
- Should throughput summarize historical event activity or current plan state?
- Should metrics write failures be fully silent, logged at debug level, or returned in tool diagnostics?

## Suggested Implementation Checklist

- Update the file inventory to include `src/sagtask/handlers/__init__.py`.
- Decide whether metrics are git-tracked or git-ignored; update `.gitignore` behavior accordingly.
- Clarify or remove execution-time tracking from scope.
- Make subtask metrics idempotent by recording transitions or aggregating latest state per subtask.
- Define a clear owner for metric file path resolution.
- Add tests for tool registration, empty metrics, malformed JSONL lines, repeated subtask status updates, coverage parsing, and context injection.
